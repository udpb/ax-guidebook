PRINCIPLE
AX 컨설팅 3가지 원칙
아래는 AX의 정의가 아니라 underdogs의 AX 컨설팅 철학이다.

원칙 1

문제는 기술이 아니라
접근 방식이다

DX가 실패하는 이유는 시스템 도입 자체가 목표가 되기 때문이다. "어떤 시스템을 쓸까"가 아니라 "어떤 병목을 풀까"가 먼저.

원칙 2

복잡한 것을 복잡하게
풀면 실패한다

SKU 150개를 관리하는 시스템을 만드는 게 아니라 "150개가 왜 필요한지"를 묻는다. 복잡성 자체가 비용이다.

원칙 3

담당자가 직접 운영할 수 있어야
진짜 전환이다

외부가 만들어준 시스템은 결국 방치된다. 실무자가 이해하고 운영할 수 있는 단순함으로 설계하고, 떠나는 것이다.

ROLE
AX 컨설턴트의 역할 — 제작이 아닌 기획, 도구가 아닌 관점

01. 인터뷰 및 기획

AX 진단 도구(CTS, ERRC-X, VRIO 등)를 활용하여 논리와 데이터로 접근

03. AI 도구 선별 및 조합

고객 상황에 맞는 가성비 높은 도구를 선별하여 워크플로우를 설계

AI를 가장 잘 쓰는 사람은 프롬프트를 잘 쓰는 사람이 아니라, 문제를 잘 구조화하는 사람이다. 그것이 AX 컨설턴트다.

06낮은 변화 실행력새 시스템 도입해도 현장에 정착 안 됨현업 플로우를 무시한 Top-down 도입혁신 시도의 반복적 실패, 냉소주의

2.13 Phase 판단 — "이 기업은 지금 어디가 급한가"
Phase급한 기업의 신호핵심 질문분석 도구
1. 비용 구조공헌이익 미측정, 수기 2h+/일, 재고회전율 <4"남는 게 없다" → 어디서 얼마나 새는가CTS (숨은 비용 투시), 병목 해부
2. 매출 성장비용 안정, 재구매 <15%, ROAS 하락"성장이 멈췄다" → 어떤 고객에게 어떤 가치를ERRC-X (가치 재설계), RFM·LTV
3. 성장동력기존 사업 안정, 시장 포화, 데이터 미활용"다음이 없다" → 핵심 역량이 뭔가VRIO (역량 스캔), 비즈니스 설계도
비용을 줄여야 매출을 늘릴 여력이 생기고, 매출이 늘어야 성장에 투자할 수 있다. 3가지를 다 할 수 없다. Phase를 판단하고 하나에 집중한다.
2.2CTS 분석 — 숨은 비용을 숫자로 증명 (Phase 1)
CTS(Cost-to-Serve)는 관리회계·공급망 분야에서 사용하는 원가 배분 개념을 AX 컨설팅 맥락에 맞게 underdogs가 적용한 것이다. 고객/채널/제품 축으로 실제 비용을 배분해서 진짜 수익성을 계산한다.

SKU별·채널별 공헌이익 미측정. 전체 P&L만 보고 있어서 어디서 새는지 모른다. CTS 분석으로 적자 채널·제품 식별이 우선.
"우리 제품이 150개인데 다 관리하기 힘들어요"복잡도 비용 #2 / Phase 1▶
매출 80%는 상위 30개가 만든다. 나머지 120개의 재고+라인 전환+소량 발주 비용이 이익을 갉아먹고 있을 가능성 높음. 파레토 분석으로 정리 대상 식별.
"전부 자동화했으면 좋겠어요"측정 부재 #3▶
측정 없이 자동화부터 가면 잘못된 데이터를 빠르게 퍼뜨린다. 수동 기록(Step 1) → 기준선 확보 → 자동화 순서가 맞다.
"김 대리가 나가면 이거 아무도 못 해요"데이터 비자산화 #5▶
데이터 비자산화. 개인 PC와 머릿속에 묶인 지식을 에이전트 스펙 카드와 운영 매뉴얼로 꺼내서 시스템에 심어야 한다.
"엑셀 파일이 너무 많아서 어떤 게 최신인지 모르겠어요"데이터 파편화 #1▶
SSOT(Single Source of Truth) 부재. 데이터가 여러 곳에 흩어져 있으면 어떤 숫자도 신뢰할 수 없게 된다. 통합 데이터 소스 구축이 모든 것의 전제.
"보고서 만드는 데 하루가 걸려요"데이터 파편화 #1 / 복잡도 비용 #2▶
반복 수작업이 자동화되지 않은 상태. 데이터 취합→정리→시각화 전 과정이 수동이다. Before/After 프로세스 맵으로 자동화 범위를 먼저 정의.
"시스템 도입했는데 현장이 안 써요"낮은 변화 실행력 #6▶
낮은 변화 실행력의 전형. 실무자 플로우를 무시한 Top-down 도입이다. 실무자와 함께 Before/After를 설계하고 동의를 받는 과정이 빠진 것.
"대표님만 데이터 보시고, 우리는 잘 몰라요"느린 의사결정 #4▶
대표 의존형 의사결정 구조. 데이터가 특정인에게 집중되면 현장 판단 속도가 떨어지고 대표 병목이 생긴다. 대시보드와 경보를 현장 담당자 단위로 설계해야 한다.
"전에 ERP 도입하려다 포기했어요"낮은 변화 실행력 #6▶
변화 시도 실패 경험 → 조직 내 냉소주의 잠재. 큰 시스템보다 Quick Win 하나로 신뢰를 먼저 쌓아야 다음 변화가 가능하다.
"재고가 얼마나 있는지 정확히 몰라요"데이터 파편화 #1 / 측정 부재 #3▶
재고 SSOT 부재. 완제품 재고(운영팀 엑셀)와 원재료 재고(공장 엑셀)가 연결되지 않은 경우가 대부분. 품절→검색순위 하락→매출 손실로 이어지는 구조.
2.4ERRC-X — AI 기반 가치 재설계 (Phase 2)
ERRC-X는 김위찬·르네 마보안의 블루오션 전략(Blue Ocean Strategy, 2005)에서 제시한 ERRC 프레임워크(제거-축소-강화-창조)를 AI 실행 관점으로 underdogs가 재구성한 것이다.
축질문예시AI(X축) 실행 방법
제거관행적이지만 가치 없는 것회전율 30% 미만 SKU 80개 단종자동 판별: AI가 "최근 90일 판매 0건, 재고 비용 月 ___만원" 리스트 자동 생성
축소과도하게 제공되는 것전 SKU 동일 패키징 → 상위 10개만 프리미엄비용 시뮬레이션: "프리미엄 적용 시 절감액 ___원" 시나리오 자동 생성
강화고객이 원하지만 부족한 것성분 투명성·사용 후기 콘텐츠 강화리뷰 분석: 전 채널 리뷰 수집 → 긍정/부정 키워드 추출 → 주간 리포트
신규업계에 없는 새 가치피부 타입별 루틴 추천개인화 엔진: 구매 이력 + 설문 데이터 기반 AI 추천. 규칙 기반 → ML 전환
ERRC-X에서 X(AI)는 "AI를 쓴다"가 아니라 "전략을 실행 가능하게 만드는 구체적 자동화"다. "어떤 데이터를 → 어떤 도구로 → 어떤 형태로 → 누가 보고 → 무엇을 판단하는가"까지 정의해야 AX다.

5.5실패 사례 라이브러리 — "이렇게 하면 안 된다"
실패 패턴증상회피 방법
측정 없이 자동화부터자동화 구축했는데 "효과가 있는지 모르겠다". 잘못된 데이터를 빠르게 퍼뜨림.Phase 판단 먼저. 2주 수동 기록(Step 1)이 기준선을 만든다.
대시보드만 만들고 끝2주 후 아무도 안 본다. 데이터도 갱신 안 됨.대시보드 + 경보는 세트. 경보가 와야 대시보드를 열게 된다.
한번에 다 바꾸려고4가지 동시 착수 → 4주 후 하나도 완성 안 됨.기초공사 최대 2개. 완성도 높은 2개가 미완성 5개보다 낫다.
우리만 쓸 수 있는 구조종료 1개월 후 "어떻게 수정해요?" 연락. 유지보수 대행 전락.인수인계 3종 세트: 에이전트 스펙 카드 + 프롬프트 라이브러리 + 운영 매뉴얼.
대표만 설득하고 끝대표는 만족, 실무자는 안 쓴다. "기존이 더 편해요."실무자와 함께 설계. Before/After를 같이 만들고 동의를 받는다.
자립 기준 미설정컨설팅 종료 조건이 없어 무한 의존 지속 또는 갑작스러운 계약 종료.컨설팅 시작 전 자립 기준 3가지를 클라이언트와 서면으로 합의.
AI 정확도 100% 기대"AI가 틀려요" → 실패로 간주. 또는 검수 없이 그대로 실행.85~95% 자동화, 나머지는 사람. 이 기대치를 킥오프에서 합의.
데이터 문제를 프롬프트로 해결AI 결과 이상한데 프롬프트만 한 달째 수정.에러의 70%는 데이터 문제. 입력 형식 → 누락/변경 → 그래도 안 되면 프롬프트.
실패 사례는 계속 추가한다. 프로젝트마다 "이번에 뭘 잘못했고, 다음에 어떻게 피하는가"를 기록한다. 이것이 쌓이면 underdogs AX의 진짜 자산.
